# Phase2 项目主线状态门禁

> 用于约束 Mock 原型中的前端按钮、后端接口和测试用例。任何入口都应遵循同一张表，不再各页面自行判断。

| 阶段 | 允许操作 | 门禁条件 | 成功后状态 | 失败/退回后状态 |
|---|---|---|---|---|
| 项目立项 | 创建项目 | Phase1 合同已签 | `pending_proposal` | 保持合同态 |
| 开题报告 | 创建/编辑/提交 | 项目存在且未完成 | `submitted` | `draft` |
| 开题审核 | 通过/驳回 | 报告为 `submitted` | 通过后生成里程碑、同步首次团队；项目进入 `ongoing` | `rejected`，允许编辑后重新提交 |
| 项目计划 | 保存草稿/生效 | 开题审核通过 | `active`；同项目旧 active 计划归档 | `draft` |
| 任务派发 | 逐任务派发 | 计划已生效、递归前置任务完成、任务为 `pending_dispatch` | `dispatched` | 保持 `pending_dispatch` |
| 任务执行 | 开始/提交审核/取消 | 已派发或整改退回 | `in_progress` / `pending_review` / `cancelled` | 整改回 `in_progress` |
| 成果物 | 上传/提交审核 | 任务存在且未取消 | 成果物 `pending`，同名旧版归档 | 旧版不可提交 |
| 成果审核 | 通过/驳回 | 任务为 `pending_review` | 通过后任务 `completed` | 驳回生成整改单，任务回 `in_progress` |
| 整改单 | 开始/提交复核/复核关闭 | 整改单未关闭 | `rectifying` → `reviewing` → `closed` | `returned`，继续整改 |
| 里程碑 | 验收通过/撤销 | 前置里程碑已通过 | `passed` | 撤销后回 `pending_acceptance` |
| 中间验收 | 发起验收 | 至少一个里程碑已通过 | 记录验收历史，项目状态不完成 | 有条件/不通过生成整改 |
| 最终验收 | 发起验收 | 全部任务完成、全部里程碑通过 | 通过后项目 `completed` | 有条件通过回 `pending_acceptance`；不通过回 `ongoing` 并生成整改 |
| 项目完成 | 展示完成结果 | 存在未撤销的最终验收通过记录 | `completed` | 无独立绕过入口 |

## 产品口径

- “项目完成”不是一个可绕过流程的主操作，而是最终验收通过后的结果确认。
- “有条件通过”和“不通过”必须产生可追踪的整改项，并允许整改后复验。
- 甘特图的状态筛选应定位到任务粒度，同时保留项目/计划/阶段父级路径。
- 所有状态枚举必须使用 `option/dict.js` 字典展示，不直接暴露英文值。
